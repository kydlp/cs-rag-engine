# このプロジェクトを CS・ナレッジマネジメントの視点から

> カスタマーサポート / カスタマーエクスペリエンス / ナレッジマネジメント / 業務改善の
> 採用担当・マネージャーの方へ。
>
> `cs-rag-engine` はエンジニア向けの README で説明していますが、中身は
> **「CS の回答品質とナレッジを、"仕組み"でどう良くするか」** の実践です。
> このドキュメントは、同じプロジェクトを **CS の言葉** で語り直したものです。

## はじめに（正直な前置き）

これは、実際に受注した **D2C スイーツブランドのカスタマーサポート自動化** の、匿名化した公開版です。
実社名・顧客データ・本番の文面は伏せていますが、設計思想・ナレッジ運用・品質評価の仕組みは忠実に再現しています。

先に正直に書いておきます。これは **個人で構築した1案件** であり、
「CS 部門を長年マネジメントした経験」や「大規模チームの運用」の代わりにはなりません。
このドキュメントが示せるのは、**CS の品質とナレッジを"感覚"でなく"仕組みと数値"で改善する考え方と、それを実際に動かした実行力** です。
そこだけは、盛らずに読んでいただけたらと思います。

---

## なぜ CS の品質にこだわるのか（原体験）

きっかけは、友人の会社のカスタマーサポートが、正直グダグダだったことです。

問い合わせが来る → スタッフが返信案を作る → その案を社内チャットに貼る → 決裁者がゴーサインを出す → やっと返信する。
この手動ループが回っていました。**返信が遅れ、担当者に依存し、急ぎの問い合わせがチャットの中で埋もれる。**

そして一番怖かったのが **「回答のブレ」** です。
国際発送やアレルギーのような「間違えたら事故になる」質問に対して、
答える人によって・その日によって回答が変わる。つまり **人間がうろ覚えで断定してしまう（人間のハルシネーション）**。

CS の体験と現場の負荷、その両方を放置できない、と思ったのがこのプロジェクトの出発点です。

## 何を、どう解いたか（CS 課題 → 打ち手）

### 課題1: 回答のブレ／オペレーター間の品質差

**打ち手 ── ナレッジを「1か所・鮮度管理・監査可能」に。**

時期で変わる事実（販売状況・賞味期限・発送目安・定期便ルール・設置場所・署名）を、
**5つのマスタ（Single Source of Truth）** に集約しました。回答はここにある事実しか使いません。

- **鮮度管理**: 各マスタに `updated_at` / `updated_by`、期限付き告知には `valid_until` を持たせ、
  **期限切れの情報は使わず「確認します」に倒す**。古い数値（例:「発送は20〜30日」）で答える事故を構造的に防ぎます。
- **更新運用**: 運用者が変更を通知 → ナレッジを更新 → 次のデプロイで即日反映、という一方通行のフロー。
  「誰が・いつ・何を変えたか」が全部残り、`git diff` で監査できます。
- **鉄則**: マスタに無い事実は、推測せず `[要確認: …]` として残すか、人に回す。

> これは求人でいう **「オペレーター向けナレッジの定常メンテナンス」「回答内容の品質改善」** の中核そのものです。

### 課題2: 承認フローのボトルネックと返信忘れ

**打ち手 ── 定型の9割を下書き自動化し、人は「承認」に集中。ただし送信はさせない。**

生成AI（Claude）が、**ナレッジに根拠のある返信下書き** を自動で作ります。人間は下書きをレビューして送るだけ。
これで返信忘れと属人化を減らします。

一方で、**「AI が顧客へ勝手に送信する」ことだけは絶対にさせません。**
これは CS における最大のリスク管理です。「送るな」とお願いするのではなく、
**送信する機能そのものをシステムに持たせない**（＝壊れようがない）設計にしています。
異物混入・健康被害・返金・法的言及など **7カテゴリは必ず人間に回す** よう、AI に判断させる前に機械的に振り分けます。

> 求人でいう **「AI 活用によるオペレーション最適化」** を、
> **顧客リスクを増やさない形** で実現する、という発想です。

### 課題3: 品質を「感覚」でなく「数値」で管理する

**打ち手 ── 回答品質を KPI 化し、改善が回っているかを毎回測る。**

- **ゴールデンセット**: 実際の問い合わせを模した評価用データに、正解カテゴリと「人に回すべきか」のラベルを付与。
- **AI による品質採点**: 生成した下書きを、**2軸（事実がナレッジと整合しているか／ナレッジに無い事実を断定していないか）** で 0–2 点採点。人手評価を約1/10のコストで再現します。
- **worst-5 改善ループ**: 最低スコアの5件の根本原因（ナレッジ不足／指示不足／分類ミス）を特定 → 直す → 再評価して悪化していないか確認。
- **進捗トラッカー**: 評価のたびに見出し指標を台帳に残し、**前回比** を出します（`reports/progress.md`）。

実際の改善（本番データ）:

```
前回比
  回答品質の合格率 : 73.5% → 80.0%   ▲ +6.5pt
  事実整合(0-2)    : 1.67 → 1.77      ▲ +0.10
  危険な見逃し(FN) : 0 → 0            変化なし（常にゼロを維持）
```

> 求人の歓迎要件 **「定量的な指標の向上に従事した経験」** に、そのまま対応します。
> 「良くなった気がする」ではなく、**数値で前回比を示せる** ことを重視しています。

## 求人要件 → このプロジェクトでの実証（対応表）

CS / CX / ナレッジマネジメント職で共通して問われる観点に沿って整理します。

| 求められる観点 | このプロジェクトでの実証 | 補足 |
|---|---|---|
| オペレーター向けナレッジの定常メンテ | 5マスタの SSOT 運用（通知→更新→即日反映）、`updated_at`/`valid_until` による鮮度管理 | 「誰がいつ何を変えたか」を監査可能に |
| 回答内容の品質改善 | AIによる2軸採点で品質を KPI 化、合格率 **73%→80%** | 主観でなく数値で改善を追跡 |
| 生成AI・AIチャットの導入検討/推進 | Claude を CS に導入。ナレッジに縛る（グラウンディング）＋安全設計 | 「導入したが事故る」を設計で回避 |
| AI活用によるオペレーション最適化 | 定型9割を下書き自動化、人は承認に集中 | 送信はさせない＝リスクは増やさない |
| 定量指標の向上に従事 | 進捗トラッカーで前回比を可視化、エスカレ精度 **98%**・危険な見逃し **0件** | `reports/progress.md` に履歴を蓄積 |
| 未整備な環境での推進 | 現場の想定外（メール導線の取り違え等）を作りながら解決 | 下記「泥臭かったこと」参照 |
| 他部署・ステークホルダーの巻き込み | 意思決定者・CS担当と、承認フローとナレッジ更新ルールを合意して運用に乗せた | ※チームのマネジメントではなく、関係者横断の調整 |
| 顧客の自己解決（FAQ/チャット） | 現状はオペレーター支援が中心。**顧客直結の FAQ/AIチャットは今後やりたいこと** | まさに求人で取り組みたい領域 |

## 泥臭かったこと（未整備な環境での実際）

きれいな一本道ではありませんでした。特にメールの導線で何度も詰まりました。

- **受信するメールアドレスを取り違えていた**: 最初は片方のアドレスを見ていたが、実際の問い合わせはもう片方に届いており、
  「本番を始めたのに問い合わせをほとんど拾えていない」状態に一度なった。転送設定で解決。
- **送信元が顧客名ではなかった**: 転送で流れてくる問い合わせの送り主が、自動送信用アドレスだった。
  そこで **本文の中に埋もれている顧客名・連絡先・注文番号を読み取って使う** 方式に切り替えた。ここは今も泥臭さが残る。

仕様書ではなく現場のログから学ぶしかない、という状況をポジティブに面白がれるタイプです。

## CS の現場から一般化できると思っている知見

1. **ナレッジは「1か所・鮮度管理・監査可能」が品質の8割。** 分散した FAQ とベテランの記憶に依存する限り、回答は必ずブレる。
2. **品質は数値化して初めて改善が回る。** 「合格率」「危険な見逃し」を定義した瞬間に、議論が"感想"から"改善"に変わる。
3. **自動化は"送信"の一歩手前で止める。** 顧客に届く最後の一押しは人間に残し、リスクは仕組み側で消す。
4. **未整備な環境は、仕様書より現場ログから学ぶ。** 実際の問い合わせを構造化することが、最良のナレッジの出発点になる。

## 正直な限界と、これからやりたいこと

- **顧客の自己解決（FAQ・AIチャット）への直結**は、このプロジェクトではまだオペレーター支援止まり。
  蓄積したナレッジ品質の土台を、顧客が直接触れる自己解決体験まで広げたい。
- **学習ループの強度**: 「人間が下書きをどう直して送ったか」を正解シグナルとして自動で還元する仕組みは、まだ半自動。
- **エスカレーションの導線**: 人に回す判定はできても、その先の受け渡しをもっと運用に耐える形にしたい。

── これらは、まさに **CS / ナレッジマネジメントのポジションで取り組みたいこと** そのものです。

## 技術用語 ↔ CS 用語の対応（読み替え表）

| リポジトリの用語 | CS の言葉での意味 |
|---|---|
| 5つのJSONマスタ / SSOT | ナレッジの一元管理（事実の唯一の出典） |

| エスカレーション（7カテゴリ） | 人間対応が必須な問い合わせの振り分け・顧客リスク管理 |
| LLM-as-judge / 合格率 | 回答品質の KPI 化と継続採点 |
| 進捗トラッカー / 前回比 | 品質指標の改善トラッキング |
| グラウンディング | 「ナレッジに書いてあることしか答えない」制御 |
