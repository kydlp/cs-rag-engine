# Safety rules — Hard Rule + Required Escalations

The system has two layers of safety enforcement: **Hard Rule** (always-on; never bypassed) and **Required Escalation** (category-driven routing to a human). Both are enforced at the code level, not in the prompt — so a model swap can never break the contract.

---

## Hard Rule (top of CLAUDE.md / production policy)

### 1. AI does not auto-send

- The engine codebase **has no `sendMessage` call**. The only Gmail write API used is `createDraft`. A model that "wants" to send cannot — there is no code path.
- For the LINE adapter, the relevant MCP tool (`send_message`) is **only invoked from the operator-facing slash command after an explicit human approval**. The engine itself never calls it.

### 2. PII masking on outbound LLM calls

- Order IDs, addresses, phone numbers, email addresses are masked before being sent to the LLM as context.
- The eval rubric explicitly counts unmasked PII in the draft as a no_hallucination=0 violation.

### 3. Required escalations (7 categories)

The categories below are **always** routed to human approval — the AI may produce a reference draft, but `escalate=true` is enforced and `decideSend()` will not allow `auto_send`:

| # | category | reason |
|---|---|---|
| 1 | Vending machine installation request | Business decision — operator chooses |
| 2 | Foreign object / health complaint | Photo required, refund/refund-flow is human-only |
| 3 | Coupon / refund / bank transfer | Monetary decision is human-only |
| 4 | Legal mention / strong complaint | Apology template only, then human takes over |
| 5 | Duplicate shipping — final decision | Receive-OK draft can be produced, but return-handling is human-only |
| 6 | Gifting / PR inquiry | Routed to a business contact |
| 7 | Payment-processor specifics | Account-level details only the operator can verify |

These mappings live in `src/engine/escalation.ts` (`HARD_ESCALATION`, `ALWAYS_ESCALATE`).

### 4. Fixed signature is generated by code, not the LLM

- The LLM system prompt explicitly **forbids** the LLM from generating a signature.
- After the LLM finishes, the deterministic code path concatenates `data/kb/master_signature.json`'s fields (`signoff` → separator → company → address → separator).
- This means an LLM cannot rewrite the company name, address, or signoff. A model that hallucinates a fake company is impossible by construction.
- Instagram DMs skip the signature when `instagram_omit=true` (set in `master_signature.json`).

### 5. KB-only facts (no invention)

The LLM system prompt:

> 事実は『提供された事実・テンプレ』のみを使用する。記載のない事実（在庫・日付・価格・住所・注文番号・追跡番号）は推測・創作しない。

If the KB doesn't say it, the model must say "[要確認: …]" or escalate. The eval rubric scores hallucinated facts as `no_hallucination=0` and surfaces them as worst-cases.

### 6. Author-facing placeholders are stripped

Template `ux_enhanced` blocks may contain author-facing placeholders like `{在庫がある場合：…}` — these never leak to a customer because compose.ts strips them via regex before sending the body to Gmail.

---

## Why code-based, not prompt-based?

A common failure mode in LLM-driven CS is: a new model release "tightens" or "loosens" a prompt rule unpredictably, and the agreement-rate metric silently slips. Our enforcement:

1. **Sends are blocked at the codebase level** (no `sendMessage` import in the engine), so a model upgrade cannot introduce sends.
2. **Escalation routing is regex-based on the customer message + classifier category**, not "ask the model to flag risky messages". The categories that must escalate are listed explicitly in `escalation.ts`.
3. **Signatures and time-dependent facts are inserted by code**, not regenerated each call, so they cannot drift with prompt changes.

Result: the production team can swap to a newer Claude / cheaper model / fine-tune without re-validating the Hard Rule.

## Auto-send allowlist (the gate)

`src/engine/gate.ts` defines a tight allowlist of categories that are *eligible* for auto-send (but only if shadow=false and all other conditions are met):

```ts
DEFAULT_AUTO_SEND_CATEGORIES = {
  "product_inquiry",   // shelf life, storage, ingredients
  "flavor_inquiry",    // which flavors are selling
  "vending_location",  // active sites
}
```

Categories not in this set (`payment`, `subscription_change`, `address_change_*`, `shipping_delay`, `receipt`, `duplicate_shipping`, `email_missing`, `other`) are always `human_approval` even when shadow is off.

The allowlist expands per-tenant after a category has shown >95% AI–human agreement on the audit log for 90 days.

---

## Shadow mode (the safe default)

`shadow=true` (default) means: even when the gate would predict `auto_send`, the engine writes to Gmail Drafts and logs `predictedMode=auto_send`. No actual send happens.

This lets the production team compare the AI's predicted send against the human's actual send and decide when to flip a category out of shadow. The audit log lives in `reports/shadow_log.jsonl`.

A row looks like:

```json
{"ts":"2026-06-13T03:00:00Z","threadId":"...","category":"product_inquiry","escalate":false,"confidence":0.85,"predictedMode":"auto_send","effectiveMode":"human_approval","gateReasons":["low-risk, high-confidence, KB-backed (product_inquiry, conf=0.85) → auto_send","shadow mode: do not auto-send; record predicted action and route to human approval"]}
```

Run `npm run eval -- --gate` (planned) or `node src/eval/gate_report.ts` for an aggregate view of the shadow-mode predictions on the golden set.
